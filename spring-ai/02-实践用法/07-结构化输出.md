# 实践 07：结构化输出 → Java DTO

读完这一篇你能：用 `chatClient.prompt(...).call().entity(MyDto.class)` 直接拿到 Java 对象；处理嵌套 record 和 `List<MyDto>`；在支持 OpenAI Structured Output 的模型上启用原生 JSON Schema 模式；加 `StructuredOutputValidationAdvisor` 做后置校验和自动重试。

## 前置依赖

只需要 OpenAI starter，Jackson 是 Spring Boot 自带依赖。

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.ai</groupId>
        <artifactId>spring-ai-starter-model-openai</artifactId>
    </dependency>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
</dependencies>
```

```yaml
spring:
  ai:
    openai:
      api-key: ${OPENAI_API_KEY}
      chat:
        options:
          model: gpt-4o-mini
```

只要项目里有一个 `ChatClient.Builder` bean（starter 自动给出），下面所有代码都能跑。

## 简单 record：从一段文本抽取人物

定义目标类型用 `record` 最干净：record 的 component 名直接成为 JSON 字段名，构造器自动配齐，反序列化也能直接走。

```java
public record Person(String name, int age, String city) {}
```

```java
@RestController
@RequestMapping("/extract")
public class PersonExtractionController {

    private final ChatClient chatClient;

    PersonExtractionController(ChatClient.Builder builder) {
        this.chatClient = builder.build();
    }

    @PostMapping
    public Person extract(@RequestBody String text) {
        return chatClient.prompt()
            .user(u -> u.text("从下面这段话里抽取人物信息：\n{text}").param("text", text))
            .call()
            .entity(Person.class);
    }
}
```

调用：

```bash
curl -X POST http://localhost:8080/extract \
  -H 'Content-Type: text/plain' \
  -d '老李今年 42 岁，住在杭州'
```

返回：

```json
{"name":"老李","age":42,"city":"杭州"}
```

`entity(Person.class)` 背后做了三件事：用 [Victools JSON Schema Generator](https://github.com/victools/jsonschema-generator) 推导出 `Person` 的 JSON Schema；把 schema 拼成"请按以下 schema 输出 JSON"格式说明，附在 user message 末尾；模型返回后用 Jackson 反序列化成对象。Schema 推导只在 `BeanOutputConverter` 构造时做一次，重复调用零开销。

## 嵌套 record：模型旅行计划

嵌套 record 一样能跑，schema 推导是递归的。

```java
public record TripPlan(
    String city,
    int days,
    List<Activity> activities) {}

public record Activity(
    String name,
    String description,
    @JsonProperty("estimated_cost_usd") double estimatedCostUsd) {}
```

```java
@GetMapping("/trip")
public TripPlan plan(@RequestParam String city, @RequestParam(defaultValue = "3") int days) {
    return chatClient.prompt()
        .user(u -> u.text("帮我规划一次 {days} 天的 {city} 旅行").param("city", city).param("days", days))
        .call()
        .entity(TripPlan.class);
}
```

如果想更精准地控制字段含义，可以加 `@JsonClassDescription` 和 `@JsonPropertyDescription`——这些注解会出现在 schema 的 `description` 里，相当于给模型贴的提示。

```java
@JsonClassDescription("一日的旅行活动安排")
public record Activity(
    @JsonPropertyDescription("活动名称，限 8 个字以内") String name,
    @JsonPropertyDescription("详细描述") String description,
    @JsonPropertyDescription("估计花费，单位美元") @JsonProperty("estimated_cost_usd") double estimatedCostUsd
) {}
```

## List 输出：拿一组对象

`entity(Class)` 重载只接受单个类型。要拿 `List` 必须用 `ParameterizedTypeReference`，这是 Spring 用来保留泛型信息的老朋友。

```java
@GetMapping("/people")
public List<Person> randomPeople(@RequestParam(defaultValue = "3") int n) {
    return chatClient.prompt()
        .user("请编造 " + n + " 个虚构人物")
        .call()
        .entity(new ParameterizedTypeReference<List<Person>>() {});
}
```

`Map<String, Foo>` / `Page<Foo>` 等参数化类型同样这么写。

## 启用 native structured output

到目前为止，schema 是通过 prompt 注入告诉模型的——这种"说服模型听话"的方式可能在简单场景下偶尔失败（多余空格、markdown 包裹）。OpenAI 的 `gpt-4o-2024-08-06` 及之后的模型（含 `gpt-4o-mini`）支持 **Structured Outputs**：把 schema 直接挂在 API 请求的 `response_format` 字段里，由 OpenAI 服务端做约束解码（constrained decoding），保证输出 100% 符合 schema。

Spring AI 在 `ChatModelCallAdvisor.augmentWithFormatInstructions` 里写了一段判断：当 context 标 `STRUCTURED_OUTPUT_NATIVE = true` 且当前 `ChatOptions` 实现了 `StructuredOutputChatOptions` 时，把 schema 写进 options 而不是 prompt（[源码](../01-源码剖析/06-结构化输出.md) 详解）。开关定义在：

```text
spring-ai-client-chat/.../ChatClientAttributes.java:33
spring-ai-client-chat/.../AdvisorParams.java:35
```

最简单的写法：用框架准备好的 advisor param 帮手。

```java
import org.springframework.ai.chat.client.AdvisorParams;

@GetMapping("/trip-native")
public TripPlan planNative(@RequestParam String city) {
    return chatClient.prompt()
        .advisors(AdvisorParams.ENABLE_NATIVE_STRUCTURED_OUTPUT)
        .user("帮我规划一次 3 天的 " + city + " 旅行")
        .call()
        .entity(TripPlan.class);
}
```

或者完全绕过框架，直接给 OpenAI 选项里塞 schema：

```java
import org.springframework.ai.openai.OpenAiChatOptions;
import org.springframework.ai.openai.OpenAiChatModel;

String schema = new BeanOutputConverter<>(TripPlan.class).getJsonSchema();

ChatResponse resp = chatClient.prompt()
    .options(OpenAiChatOptions.builder()
        .responseFormat(OpenAiChatModel.ResponseFormat.builder().jsonSchema(schema).build())
        .build())
    .user("帮我规划一次 3 天的杭州旅行")
    .call()
    .chatResponse();
```

在 `OpenAiChatModel` 里这条分支会被翻译成官方 SDK 的 `ResponseFormatJsonSchema`（[OpenAiChatModel.java:878-893](../../../spring-ai/models/spring-ai-openai/src/main/java/org/springframework/ai/openai/OpenAiChatModel.java)），并自动设 `strict: true`。

注意点：

- 不是所有 OpenAI 模型都支持 native 模式，`gpt-3.5-*` 走老的 prompt 注入路径
- `strict: true` 模式下，schema 不能出现 OpenAI 不支持的关键字（如 `oneOf`、`pattern`、`format`），否则 API 直接 400
- Anthropic / Bedrock Converse 目前没有等价的"挂 schema"通道，开了 `ENABLE_NATIVE_STRUCTURED_OUTPUT` 也只会被忽略（`StructuredOutputChatOptions` instanceof 判断不通过）

## 校验 + 自动重试

prompt 注入模式仍然可能失败，比如模型偶尔多包一层 markdown、嵌套字段类型出错。`StructuredOutputValidationAdvisor` 的做法是：拿到 LLM 输出后用 [networknt/json-schema-validator](https://github.com/networknt/json-schema-validator) 校验 JSON，不通过就把错误信息追加到 user message 重发。

```java
import org.springframework.ai.chat.client.advisor.StructuredOutputValidationAdvisor;
import org.springframework.core.ParameterizedTypeReference;

var validator = StructuredOutputValidationAdvisor.builder()
    .outputType(new ParameterizedTypeReference<TripPlan>() {})
    .maxRepeatAttempts(2)
    .build();

TripPlan plan = chatClient.prompt()
    .advisors(validator)
    .user("帮我规划一次 3 天的杭州旅行")
    .call()
    .entity(TripPlan.class);
```

它的循环逻辑（看 [StructuredOutputValidationAdvisor.java:124-178](../../../spring-ai/spring-ai-client-chat/src/main/java/org/springframework/ai/chat/client/advisor/StructuredOutputValidationAdvisor.java)）大致是：

```text
do {
    response = chain.copy(this).nextCall(request)
    if (!hasToolCalls && !validate(response)) {
        request = request + "Output JSON validation failed because of: ..."
        continue
    }
    break
} while (count <= maxRepeatAttempts)
```

注意它用 `chain.copy(this)` 复制子链重入——这正是第 4 篇 advisor 链 deque 设计的应用场景。

要点：

- 这个 advisor **不支持流式**，对 `stream()` 调用会抛 `UnsupportedOperationException`
- `maxRepeatAttempts` 默认 3。失败次数到了不会抛异常，会把最后一次（仍不合规的）响应交给上层 —— 也就是 `entity()` 反序列化时还是可能炸
- 与 native structured output 同时启用没意义：native 模式由模型保证合规，validator 只会做无谓校验

## 踩坑

**1. 模型把 JSON 包在 markdown 里**

`Do not include markdown code blocks in your response.` 这一行就是写在 `BeanOutputConverter.getFormat()` 模板里的（[BeanOutputConverter.java:266-276](../../../spring-ai/spring-ai-model/src/main/java/org/springframework/ai/converter/BeanOutputConverter.java)）。即便如此，部分模型仍会输出 ` ```json ... ``` `。`BeanOutputConverter` 内部有一个 `MarkdownCodeExtractor` 兜底剥壳，正常情况下用户感知不到。但如果你自己用 `StructuredOutputConverter`，记得用框架默认实例。

**2. Java 字段类型推导出错**

最常见的：用 `Date` / `LocalDate`，schema 推出来是 `string` 但模型返回 ISO-8601 字符串后 Jackson 反序列化失败。建议直接用 `String` 字段，自己 parse；或者注册自定义 `JsonDeserializer`。

**3. 嵌套层级太深**

OpenAI 的 native structured output 文档要求最多 5 层嵌套、字段总数 ≤ 100。超出会报 schema 错误。这种情况下要么扁平化你的领域模型，要么退回 prompt 注入模式。

**4. 中文字段名不行**

JSON Schema generator 取的是 record component 名（即 Java 字段名），所以字段名必须是 ASCII。需要中文字段输出时用 `@JsonProperty("中文字段")` 重命名。

## 完整示例

把上面所有零件拼成一个 `@RestController`：

```java
@RestController
@RequestMapping("/extract")
public class StructuredOutputController {

    private final ChatClient chatClient;
    private final StructuredOutputValidationAdvisor tripValidator;

    StructuredOutputController(ChatClient.Builder builder) {
        this.chatClient = builder.build();
        this.tripValidator = StructuredOutputValidationAdvisor.builder()
            .outputType(new ParameterizedTypeReference<TripPlan>() {})
            .maxRepeatAttempts(2)
            .build();
    }

    @PostMapping("/person")
    public Person person(@RequestBody String text) {
        return chatClient.prompt()
            .user(u -> u.text("抽取人物：\n{t}").param("t", text))
            .call()
            .entity(Person.class);
    }

    @GetMapping("/trip")
    public TripPlan trip(@RequestParam String city) {
        return chatClient.prompt()
            .advisors(AdvisorParams.ENABLE_NATIVE_STRUCTURED_OUTPUT, tripValidator)
            .user("帮我规划一次 3 天的 " + city + " 旅行")
            .call()
            .entity(TripPlan.class);
    }

    @GetMapping("/people")
    public List<Person> people() {
        return chatClient.prompt()
            .user("请编造 5 个虚构人物")
            .call()
            .entity(new ParameterizedTypeReference<List<Person>>() {});
    }
}
```

跑起来：

```bash
curl http://localhost:8080/extract/trip?city=杭州 | jq
curl http://localhost:8080/extract/people | jq
curl -X POST http://localhost:8080/extract/person -H 'Content-Type: text/plain' -d '小王 28 岁，北京'
```
