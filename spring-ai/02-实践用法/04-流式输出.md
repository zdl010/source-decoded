# 实践 04 · 流式输出 SSE

跑完这一篇，你能：把 `chatClient.prompt().stream().content()` 拿到的 `Flux<String>` 直接转成 SSE 推到前端，前端用浏览器原生 `EventSource` 边接收边渲染；并且能在中途断开、超时、上游报错时控制好链路。

剖析侧的细节（reactor context、终结 advisor 的 `Flux.deferContextual`）见 [01-源码剖析/02-调用旅程](../01-源码剖析/02-调用旅程.md) 流式分支一节，本篇只关心怎么把它接出来用。

## 一、先在 main 里看一眼 Flux

在写 Controller 之前，先用最小代码确认流是真在边吐边到的。

```java
@SpringBootApplication
public class StreamingDemo {
    public static void main(String[] args) {
        ConfigurableApplicationContext ctx = SpringApplication.run(StreamingDemo.class, args);
        ChatClient chatClient = ctx.getBean(ChatClient.Builder.class).build();

        chatClient.prompt()
                .user("用三段话讲一下 SSE 协议的特点")
                .stream()
                .content()                  // Flux<String>
                .doOnNext(System.out::print)
                .blockLast();               // demo 用，生产不要 block
    }
}
```

跑起来你会看到字一段一段冒出来，每段就是一个 chunk。`stream()` 接口和 `call()` 平行，签名定义在 `spring-ai-client-chat/.../ChatClient.java:183`：

```java
interface StreamResponseSpec {
    Flux<ChatResponse> chatResponse();      // 完整结构，含 metadata
    Flux<String> content();                 // 只要文字
}
```

## 二、最小 SSE Controller

WebFlux 下，把 `Flux<String>` 包成 `Flux<ServerSentEvent<String>>` 直接返回，框架会用 `text/event-stream` 编码：

```java
@RestController
@RequestMapping("/ai")
public class ChatStreamController {

    private final ChatClient chatClient;

    public ChatStreamController(ChatClient.Builder builder) {
        this.chatClient = builder.build();
    }

    @GetMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> stream(@RequestParam String q) {
        return chatClient.prompt()
                .user(q)
                .stream()
                .content()
                .map(chunk -> ServerSentEvent.<String>builder()
                        .data(chunk)
                        .build())
                .concatWith(Flux.just(ServerSentEvent.<String>builder()
                        .event("done")
                        .data("")
                        .build()));
    }
}
```

几个细节：

- `produces = MediaType.TEXT_EVENT_STREAM_VALUE` 一定要给，不然 Spring 默认走 `application/json` 会缓冲整个 Flux。
- 末尾追加一个 `event: done` 是给前端关闭连接用的标记，否则浏览器会一直 retry。
- `ServerSentEvent` 默认每条带 `data:` 前缀，多行 `data` 由它自己处理换行转义；不要自己拼 `data: xxx\n\n`。

启动：`./mvnw spring-boot:run`，curl 验证：

```bash
curl -N "http://localhost:8080/ai/chat/stream?q=hi"
```

`-N` 关闭 curl 自己的输出缓冲，否则你看到的是一次性吐出来的整段，会以为没流。

## 三、前端最小消费

```html
<!doctype html>
<html>
<body>
  <div id="out" style="white-space: pre-wrap; font-family: monospace"></div>
  <script>
    const out = document.getElementById('out');
    const es = new EventSource('/ai/chat/stream?q=' + encodeURIComponent('用三段话讲讲 SSE'));
    es.onmessage = (e) => { out.textContent += e.data; };
    es.addEventListener('done', () => es.close());
    es.onerror = () => { es.close(); out.textContent += '\n[stream closed]'; };
  </script>
</body>
</html>
```

关键点：

- `onmessage` 接收没有 `event:` 名字的默认事件——和上面 Controller 里的 chunk 对应。
- `addEventListener('done', ...)` 接收命名事件，收到就主动 `close()`，避免 `EventSource` 自动重连让模型再答一次。
- `onerror` 是兜底——网络断、服务器关连接都会触发；不要在这里重试，否则可能烧 token。

## 四、要 metadata 时用 chatResponse() 流

只要文字够用就走 `content()`；要看 token 消耗、`finishReason`、tool call 时走 `chatResponse()`：

```java
@GetMapping(value = "/chat/stream-full", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> streamFull(@RequestParam String q) {
    return chatClient.prompt()
            .user(q)
            .stream()
            .chatResponse()
            .map(resp -> {
                String content = resp.getResult() == null ? ""
                        : resp.getResult().getOutput().getText();
                String finish = resp.getResult() == null ? null
                        : resp.getResult().getMetadata().getFinishReason();
                ServerSentEvent.Builder<String> b = ServerSentEvent.<String>builder().data(content);
                if (finish != null) {
                    b.event(finish.toLowerCase());     // 例如 stop / tool_calls
                }
                return b.build();
            });
}
```

每个 `ChatResponse` 是一个增量 chunk，`getMetadata().getFinishReason()` 在最后一帧才非空——可以以此通知前端结束。要算 token 总数，在最后一帧读 `resp.getMetadata().getUsage()`。

## 五、错误传播与客户端断开

Flux 链路里加 `onErrorResume` 把异常转成"最后一帧错误事件"，比直接抛 500 友好——浏览器 EventSource 收到 5xx 默认会重连。

```java
return chatClient.prompt().user(q).stream().content()
        .map(chunk -> ServerSentEvent.<String>builder().data(chunk).build())
        .timeout(Duration.ofSeconds(60))
        .onErrorResume(ex -> Flux.just(
                ServerSentEvent.<String>builder()
                        .event("error")
                        .data(ex.getClass().getSimpleName() + ": " + ex.getMessage())
                        .build()))
        .doOnCancel(() -> log.info("client disconnected, upstream cancelled"));
```

- `timeout` 控制 Flux 的"两次发送之间"间隔，不是总时长——LLM 卡住时这个比 `Mono.timeout` 实用。
- `onErrorResume` 让连接以正常的 `event: error` 帧结束，前端 `addEventListener('error', ...)` 接住后主动 `close()`。
- `doOnCancel` 在浏览器关页签或 `EventSource.close()` 时触发——reactor 会沿链路向上 cancel，OpenAI 的 HTTP 连接也会被关闭，**不会继续烧 token**。这是 reactive stack 相比 SseEmitter 的最大优势。

## 六、MVC 项目的兜底写法

如果你项目仍然是 spring-boot-starter-web（Servlet）而不是 webflux，`Flux<ServerSentEvent>` 也能返回——Spring MVC 5+ 支持，但底层用阻塞线程持有连接。要能 cancel 上游，更稳妥的写法是用 `SseEmitter` + 手动 subscribe：

```java
@GetMapping("/chat/stream-mvc")
public SseEmitter streamMvc(@RequestParam String q) {
    SseEmitter emitter = new SseEmitter(0L);  // 0 = 不超时
    Disposable d = chatClient.prompt().user(q).stream().content()
            .subscribe(
                chunk -> {
                    try { emitter.send(chunk); }
                    catch (IOException e) { emitter.completeWithError(e); }
                },
                emitter::completeWithError,
                emitter::complete);
    emitter.onCompletion(d::dispose);
    emitter.onTimeout(d::dispose);
    return emitter;
}
```

注意 `emitter.onCompletion`/`onTimeout` 必须 dispose 上游 subscription，否则客户端断开时上游 Flux 还在跑。

## 七、踩坑清单

- **看不到流，是一次到齐的**：99% 是中间有反向代理（Nginx / ALB）或 CDN 在缓冲。Nginx 要加 `proxy_buffering off; proxy_cache off; proxy_read_timeout 600s;` 这三条，并确保响应头有 `X-Accel-Buffering: no`（可以在 Controller 里塞响应头）。
- **跨域 EventSource**：`EventSource` 不支持自定义 header，认证只能走 cookie 或 query string；CORS 要允许 `text/event-stream`。
- **OpenAI 兼容代理拒收**：第三方 OpenAI-compatible 服务有的不实现 SSE，得在 properties 里关掉默认流式行为或换 endpoint。
- **token 计费在结尾**：边流边发的 chunk 里 `usage` 多半为空，最后一帧才填——别在中间帧拿 0 当成事实。
- **客户端 retry 风暴**：浏览器 EventSource 默认收到任何错误都会重连。一定要在前端 `error` / `done` 事件里 `close()`，否则一次失败会被翻成 N 倍调用。

## 八、最小可跑的整套示例

把上面三块（pom.yml、Controller、index.html）放进同一个 Spring Boot 项目，把 `index.html` 丢到 `src/main/resources/static/`，`./mvnw spring-boot:run` 之后访问 `http://localhost:8080/index.html` 就能看到字逐字蹦出来的效果。整个工程文件 ≤ 50 行 Java。

## 思考题

1. 如果你想让前端展示"光标在打字"的效果（每个字符之间有 30ms 延迟），改 Controller 端用 `delayElements(Duration.ofMillis(30))` 还是改前端用 setTimeout，哪种实现下"取消"更可控？
2. 同一个 controller 同时支持 SSE 和 WebSocket 两种推送方式时，怎么把 `Flux<ChatResponse>` 复用、不让两个 endpoint 各起一次模型调用？
3. SSE 单连接是 HTTP/1.1 长连接，浏览器对同源连接数有上限（常见 6）；如果一个页面同时开 7 路对话流会发生什么？怎么规避？
