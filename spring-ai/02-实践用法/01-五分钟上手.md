# 五分钟跑通第一个 ChatClient

读完这一篇，你能得到一个最小可跑的 Spring Boot 工程：注入一个 `ChatClient`，写一个 `@RestController`，本地或对接 OpenAI 都能 `curl` 拿到模型回复。整个工程只需要 1 个 pom、1 段 yaml、1 个 Java 文件。

后面所有实践篇都会以这个最小工程为底子继续加东西。如果你想知道这一行 `chatClient.prompt("hi").call().content()` 背后调用栈走了什么路径，看 [01-源码剖析/02-调用旅程](../01-源码剖析/02-调用旅程.md)。

---

## 前置依赖

- **JDK 17 或更高**——`spring-ai-parent/pom.xml` 把 `<java.version>` 钉在 17（`pom.xml:270`）
- **Spring Boot 4.1.0-RC1**——本系列基于 `2.0.0-SNAPSHOT`，对应 Boot 4 RC（`pom.xml:276`）
  - 想要更稳定的生产组合：用 `spring-ai 1.0.x` + `Spring Boot 3.4.x`，下面的代码 90% 兼容，只需把版本号换回去
- **Maven 3.9+**（Gradle 也行，本文用 Maven）
- 可选：本地装一份 [Ollama](https://ollama.com) 来跑场景 B，无需任何 API key

│   ├── QuickstartApplication.java

### QuickstartApplication.java

```java
package com.example.quickstart;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class QuickstartApplication {
    public static void main(String[] args) {
        SpringApplication.run(QuickstartApplication.class, args);
    }
}
```

### ChatController.java

```java
package com.example.quickstart;

import org.springframework.ai.chat.client.ChatClient;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class ChatController {

    private final ChatClient chatClient;

    // ChatClient.Builder 是 prototype-scope 的 bean，
    // 见 01-源码剖析/09-boot-观测.md 讲为什么不是 singleton
    public ChatController(ChatClient.Builder builder) {
        this.chatClient = builder.build();
    }

    @GetMapping("/chat")
    public String chat(@RequestParam(defaultValue = "用一句话介绍 Spring AI") String q) {
        return chatClient.prompt(q).call().content();
    }
}
```

3 行代码就是一次完整的 LLM 调用。`prompt` 准备请求、`call` 触发同步调用、`content` 取出回复字符串。流式版本是 `.stream().content()`，第 4 篇会展开。

## 场景 A：接 OpenAI

### application.yml

```yaml
spring:
  ai:
    openai:
      api-key: ${OPENAI_API_KEY}
      chat:
        options:
          model: gpt-4o-mini
          temperature: 0.7
```

`spring.ai.openai.*` 这个前缀挂在 `OpenAiConnectionProperties.java:25`，子前缀 `spring.ai.openai.chat.*` 挂在 `OpenAiChatProperties.java:32`。`api-key` 用 `${OPENAI_API_KEY}` 占位读环境变量，避免把 key 写进代码。

### 启动

```bash
export OPENAI_API_KEY=sk-...
mvn spring-boot:run
```

启动日志里看到 `Started QuickstartApplication in X seconds` 就是好了。

### 验证

```bash
curl 'http://localhost:8080/chat?q=用一句话解释什么是 RAG'
```

预期是一句中文回复，比如：

```text
RAG（Retrieval-Augmented Generation）是把外部知识检索结果拼到 prompt 里，让模型在生成时能引用最新或私有数据的技术。
```

## 场景 B：接本地 Ollama

把 pom 里的 OpenAI starter 注释掉，把 Ollama starter 取消注释。其他 Java 代码完全不用动——这正是 `ChatModel` 抽象在做的事，详细可看 [01-源码剖析/03-chatmodel-options](../01-源码剖析/03-chatmodel-options.md)。

### 拉一个小模型

```bash
ollama serve            # 后台跑 Ollama 服务
ollama pull llama3.2:1b # 拉一个 1B 的小模型，几百 MB
```

### application.yml

```yaml
spring:
  ai:
    ollama:
      base-url: http://localhost:11434
      chat:
        options:
          model: llama3.2:1b
          temperature: 0.7
```

`base-url` 默认就是 `http://localhost:11434`（`OllamaConnectionProperties.java:36`），可以省略；写出来是为了让读者知道改远程 Ollama 时改这里。

### 启动 + 验证

```bash
mvn spring-boot:run
curl 'http://localhost:8080/chat?q=Say hi in one sentence.'
```

`llama3.2:1b` 中文不强，建议先用英文 prompt 验证链路通了再说。链路通了之后想换大模型，把 `model:` 改成 `qwen2.5:7b` 这种然后 `ollama pull` 一下即可，不用改 Java 代码。

## 踩坑速查

**`401 Unauthorized` / `OpenAiApi$OpenAiApiException`**
环境变量没设，或 yaml 里 `api-key` 写错前缀（必须是 `spring.ai.openai.api-key`）。最简单的检查：启动日志里搜 `OpenAiApi` 看 base-url 和 key 是否被加载。

**`No qualifying bean of type 'ChatClient.Builder'`**
两种情况：a) starter 没引；b) 同时引了多个 chat starter（OpenAI + Anthropic + ...）但没指定用哪个 `ChatModel`。第一种检查 pom；第二种看 [实践 02-多 Provider 切换](./02-多provider切换.md)。

**`Connection refused: /127.0.0.1:11434`**
Ollama 服务没起。`ollama serve` 或者直接打开 Ollama 桌面 app；如果端口被占用，改 `spring.ai.ollama.base-url`。

**模型名错**
OpenAI 用 `gpt-4o-mini`、`gpt-4o`、`gpt-3.5-turbo` 这种官方 ID；Ollama 用 `ollama list` 出来的本地名字。错了通常会 `404 model not found`。

**响应是空字符串**
看 chat 模型是不是被某种过滤拦了（敏感词、context length 超限），把 `temperature` 调高、prompt 改简单先验证通路。也可以在 controller 里把 `chatClient.prompt(q).call().chatResponse()` 拿到完整 `ChatResponse`，看 `result.getMetadata().getFinishReason()`——`length`/`content_filter` 这种就是被截了。

**SNAPSHOT 解析慢 / 失败**
`mvn -U` 强制刷新，或者干脆切到稳定版（spring-ai 1.0.x + Boot 3.4.x），把 `<repositories>` 段删掉。

## 这一行代码背后

回过头看 controller 里的核心三步：

```java
chatClient.prompt(q)   // 1. 拼请求
          .call()      // 2. 同步触发
          .content();  // 3. 取字符串
```

每一步都对应 spring-ai 里的一组类：

- `prompt(q)` 返回 `ChatClient.ChatClientRequestSpec`，把字符串包成 `Prompt`，走到 `DefaultChatClient.DefaultChatClientRequestSpec`（`spring-ai-client-chat/.../client/DefaultChatClient.java`）
- `call()` 触发 advisor 链（`DefaultAroundAdvisorChain.nextCall`），走完所有 advisor 之后由终结 advisor `ChatModelCallAdvisor` 真正调到 `ChatModel.call(prompt)`——这就是 OpenAI HTTP 请求发出去的地方
- `content()` 从 `ChatClientResponse` 里把 assistant 消息的纯文本抽出来

如果你对"为什么 spring-ai 要做 ChatClient + Advisor 链"这条主轴有兴趣，去读 [01-源码剖析/02-调用旅程](../01-源码剖析/02-调用旅程.md) 和 [04-advisor-链](../01-源码剖析/04-advisor-链.md)，那两篇会讲清楚这三行代码到底做了多少事。
